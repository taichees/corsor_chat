# 不動産取得費シミュレーター 開発ロードマップ

## 1. プロジェクト概要

- アプリ名: 不動産取得費シミュレーター
- 目的: 不動産取得時に発生する税金・手数料・保険・継続費用を、法令や一般的な計算根拠に基づいて概算し、初年度費用と次年度以降の費用を整理して提示する。
- 立ち位置: 単発の計算ツールではなく、ライフステージ変化時に使うツール群の1サービスとして育て、将来的には親サイト配下に統合する。

## 2. 最終到達イメージ

### 2-1. ユーザー向けの完成形

- 物件価格、種別、所在地、築年数、ローン条件などを入力すると、取得時コスト全体を一覧化できる
- 初年度に必要な現金、入居後に遅れて来る費用、毎年または継続的に発生する費用が分かれる
- 税金は計算根拠、特例条件、注意点、支払タイミングを表示できる
- 手数料や保険は概算値を出した上で、実額が分かれば手入力で補正できる
- 複数物件を比較できる
- PDFや共有リンクで家族、不動産会社、FPと共有できる

### 2-2. 事業としての完成形

- SEOで集客できる無料計算ページを複数持つ
- 住宅ローン、火災保険、引越し、相談サービスなどへ自然に送客できる
- 一部機能を有料化できる
- 将来的に不動産会社やFP向けのB2B提供へ展開できる
- 親サイトで他のライフステージ系ツールと横断導線を持てる

## 3. 開発方針

### 3-1. 基本方針

1. 最初は税金に絞る
2. 計算ロジックをUIから分離する
3. 「概算」と「補正入力」を両立する
4. 初年度費用と継続費用を必ず分離する
5. 法改正や特例期限に追随しやすい構造にする

### 3-2. 非目標

- 最初から全自治体、全金融機関、全保険商品に完全対応しない
- 最初から会員機能や決済を入れない
- 最初からB2B管理画面を作らない
- 法律相談や税務相談の代替と見える表現は避ける

## 4. 推奨プロダクト構成

### 4-1. 初期技術スタック案

- フロントエンド: Next.js + TypeScript
- UI: Tailwind CSS
- 計算ロジック: `lib/calculators/` 配下に純粋関数として分離
- コンテンツ: Markdownまたは静的ページ
- 分析: Google Analytics, Search Console, 広告タグは後から追加
- 永続化: MVPでは不要。比較保存フェーズでDB導入を検討

### 4-2. 初期ルーティング案

- `/` : LP
- `/simulator` : メイン計算画面
- `/guides` : 解説一覧
- `/guides/taxes` : 税金解説
- `/guides/fees` : 手数料解説
- `/guides/insurance` : 保険解説
- `/legal` : 注意事項、免責、計算根拠

## 8. データとロジック設計方針

- 計算式はUI内に埋め込まない
- 税率、閾値、特例期限は設定ファイルに分離する
- 物件条件に応じた分岐はテスト可能な関数で実装する
- 将来の法改正に備え、年度別設定を持てる構造にする

### 例

- `config/taxRules/2026.ts`
- `lib/calculators/stamp-tax.ts`
- `lib/calculators/registration-tax.ts`
- `lib/calculators/acquisition-tax.ts`
- `lib/formatters/currency.ts`

## 10. リスクと対策

### 10-1. 法改正リスク

- 対策: 税率と期限を設定ファイル化し、更新日を明示する

### 10-2. 概算ロジックへの不信

- 対策: 根拠、概算である理由、実額入力による補正導線を表示する

### 10-3. YMYL領域でのSEO難易度

- 対策: 計算機だけでなく根拠説明ページ、FAQ、監修導線を整える

### 10-4. スコープ肥大化

- 対策: Phase 1では税金以外を入れない

## 12. Cursorへの最初の依頼文の叩き台

以下をそのまま初回プロンプトの叩き台として使える。

> 「不動産取得費シミュレーター」というWebアプリを作ってください。  
> まずは税金MVPに絞り、印紙税・登録免許税・不動産取得税・固定資産税/都市計画税の年額目安を計算できるようにしてください。  
> Next.js + TypeScript + Tailwind で構築し、`/` にLP、`/simulator` にシミュレーター画面を用意してください。  
> 計算ロジックは `lib/calculators/` に分離し、将来の法改正に備えて設定ファイル化しやすい構造にしてください。  
> 出力では、税目別内訳、初年度費用、次年度以降にかかる費用、支払タイミング、計算根拠の注記を表示してください。  
> まずは UI とモックロジックではなく、テスト可能な純粋関数ベースの実装を優先してください。  
> また、概算であることと免責を明示してください。
